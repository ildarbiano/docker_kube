====== Проблемы разворачивания одного компонента ==============
Сервер: CPU, Memory, Network, Storage
ОС: Зависимости и Переменные окружения (Java) для Application << Наблюдаемость (monitoring-Grafana, logging-ELK)
Разграничение доступа к Application (безопасность)
Вертикальное 	масштабирование 	- добавить ресурсы (CPU, Memory, Network, Storage). Временное решение производительности.
Горизонтальное 	масштабирование		- добавление приложения и его балансировка. 	
Оптимизация ресурсов на одном сервере, где запущены applications.
Период недоступности application, в момент deploy new version.

====== Масштабирование одного компонента =======================
Load Balancer (Proxy)
Docker		- обеспечивает изоляцию работы пapplication, по используемым ресурсам;
Application оборачивается в docker image.
OpenShift	- система оркестрации;
Kubernetes	- система оркестрации;
Docker Swarm 	— система оркестрации контейнеров, встроенная в Docker. 
Docker Swarm 	- обеспечивает отказоустойчивость application;
Docker Swarm 	— позволяет управлять кластером хостов Docker как единым виртуальным хостом. 
Docker Swarm 	- основная задача — автоматизация развёртывания, масштабирования и управления контейнерными приложениями. 
Docker Swarm 	- особенности:
-	Swarm автоматически распределяет контейнеры по узлам кластера. 
-	Перезапускает упавшие сервисы. 
-	Обеспечивает балансировку нагрузки. 
-	Контролирует состояние задач и автоматически перезапускает их в случае сбоев или падений. 

Docker Swarm 	- Принцип работы:
Система организована по иерархическому принципу: 
1/	на вершине находится лидер-менеджер, выбранный через алгоритм консенсуса, который координирует действия остальных менеджеров. Рекомендуемое количество менеджеров — нечётное (обычно 3 или 5), чтобы обеспечить корректную работу механизма голосования. 
Важно: Swarm не запускает контейнеры явно — этим занимаются сервисы. Для достижения отказоустойчивости можно указать сервису количество реплик — нод, на которых он должен запустить контейнеры. Swarm найдёт подходящие хосты, запустит контейнеры и будет следить за ними. Если один из хостов выйдет из строя, создаст новую реплику на другом хосте. 


====== Microserves многокомпонентная архитектур ================
Service Discovery	-	это система, которая ведёт специальный журнал, в котором серверы и компоненты инфраструктуры сообщают о своих адресах и состоянии. 
Service Discovery 	-	позволяет реализовать взаимодействие компонентов с минимальным ручным вмешательством. Он даёт возможность обнаруживать сервисы и заносить их в базу автоматически. 
Service Discovery	-	Некоторые возможности :
-	Подключение новых приложений в уже существующее окружение. Адресация приложений в окружении, с изменением адресации, при разворачивани на другом сервере.
-	Распределение нагрузки в системе и включение новых серверов в кластер.
-	Хранение конфигов, сертификатов и списка активных серверов.
-	Обнаружение сбоев и отказов.
Manifest	- Services, Deployments. Разворачивание application сводится к построению/описание ресурсов, через манифесты;
Manifest	- отвечают за "Проблемы разворачивания application";
Manifest	- содержит:
Deployment(s)	- указываются образы контейнеров, template-шаблон, тома. позволяет: 
	создать несколько реплик (подов) приложения;
	задать необходимые для работы ресурсы (CPU, RAM);
	если на текущей ноде закончатся ресурсы, поды будут перенесены на другой узел кластера;
	параметры масштабирования. Автоматически поддерживать количество подов в кластере;
	параметры живучести application. Если отвалилась одна из реплик, Deployment автоматически поднимет другую реплику.
StatefulSet(s)	- указываются образы контейнеров. Это контроллер, предназначенный для управления состоянием приложений, которым важно сохранять идентичность и данные при перезапуске. В отличие от Deployment, который подходит для статeless-сервисов, StatefulSet используется для приложений, требующих сохранения состояния между перезапусками.
ConfigMap	- словарь конфигурации;
Ingress		- настраивает доступ внешний. Инструмент управления входящим трафиком, который позволяет организовать доступ к различным сервисам через единую точку входа. Это система правил маршрутизации HTTP и HTTPS-трафика, работающая на уровне 7 модели OSI. Для сетевого доступа к instances.s Особенности Ingress:
	маршрутизация запросов по доменным именам и путям URL;
	автоматическое управление SSL/TLS-сертификатами;
	балансировка нагрузки между различными экземплярами сервисов;
	настройка правил доступа и ограничений безопасности.
Service		- настраивает доступ изнутри cluster. Настройки Persistence Volume. Селектор переключения при Blue-green strategy of deployment. 
Cluster		- предоставляет необходимые ресурсы, для обеспечения работоспособности.
oc		- утилита управления OpenShift;
kubectl		- утилита управления Kubernetes;
helm		- утилита управления, позволяет гибко параметризировать создание манифестов и вести контроль версий релизов, для возможности отката на предыдущие версии;
Hook 		- это функция, которая позволяет вмешиваться в жизненный цикл сервиса на определённых этапах. Она может быть зарегистрирована вокруг, перед, после или при ошибках метода сервиса. Post Start Hook, Pre Stop Hook. Например:
	- хук, который будет выполняться после старта конейнера, в нём можно выполнить набор shell команд. 
	- хук, который необходимо выполнить до остановки контейнера
Hook		- цель хуков — реализовать пользовательский код в путь выполнения сервиса, не изменяя исходный код. Это помогает: 
	-повысить масштабируемость и настраиваемость программы; 
	-сделать логику сервиса гибкой и композируемой, что облегчает отслеживание и отладку. 
Readiness Probe	- это проверка готовности контейнера принимать входящий сетевой трафик и обрабатывать запросы. Она гарантирует, что контейнер завершил процесс инициализации и готов к обработке запросов. Цель:
	- Временно изолировать контейнер, который не готов обрабатывать запросы, например, во время обновления базы данных или загрузки кэша.
	- Контролировать запуск приложений с долгой инициализацией.
Readiness Probe - гарантирует, что Kubernetes не будет отправлять трафик (не распределяет, не балансирует нагрузку) до тех пор, пока приложение полностью не запустится. 
Liveness Probe	- это диагностическая проверка, которая определяет, «жив» ли контейнер, то есть работает ли он корректно. Если проверка не проходит, Kubernetes считает контейнер нерабочим и автоматически перезапускает его. 

Request	resources гарантирует достаточность ресурсов на этом узле, этой Node. Гарантирует, что не превысит возможности узла. Гарантирует предоставление ресурсов, указанных в сумме Requests.
Limit	resources при привышении имитов K7S начнёт процедуру остановки приложения. Для ограничения приложения по используемым рксурсам. Сумма лимитов может больше физических ресурсов, за захват ресурсов, в пределах лимитов, приложения конкурируют между собой.
Update stratagy	- обновление Application с контролем времени недоступности, так что потребители не заметили изменения при обновлении приложения;
Deploy strategy	- как заменить Application так, чтобы пользователи не получили пробел в обслуживании:
	Rollingupdate strategy 	- удаление одного/по одному пода/у версии №1 и создание нового с версией №2, так до полниго замещение подов на версию №2, но появляется/существует время, когда пользователь попадает, как на верси №1 application, так и на версию №2; Стратегия по-умолчанию в Deployment;
	Recreate startegy 	- одновременное удаление подов версии №1 и создание подов версии №2, но появляется время простоя application;
	Blue-Green strategy 	- создаётся новый deployment и переключается трафик на новый, а старый удаляется. Переключение происходит через селектор Service;
	Canary strategy 	- и повая версия и старая версия, пока не убедимся в работоспособности новой версии. Настройка балансировщика нагрузки, чтобы он переводил пользователей (лояльных) на новую версию. 
Istio (Service Mesh) 		— это платформа для управления сетевым взаимодействием микросервисов в распределённых системах. Она обеспечивает единообразный способ подключения, мониторинга и защиты микросервисов, при этом не требует изменений в коде приложений. 
Istio 				- работает как sidecar-прокси — контейнер, который запускается вместе с каждым сервисом в виде отдельного контейнера (sidecar). Этот контейнер обрабатывает весь входящий и исходящий трафик сервиса, выполняя задачи, связанные с управлением сетью и безопасностью. Функции. Некоторые возможности Istio:
	- Управление трафиком: маршрутизация запросов, балансировка нагрузки, отказоустойчивость. Например, распределение трафика между сервисами или канареечное развёртывание, когда только часть клиентского трафика отправляется на новую версию приложения. 
	- Наблюдаемость: мониторинг производительности сервисов, сбор метрик, трассировка, журналирование. 
	- Отказоустойчивость: механизм таймаутов, повторных попыток выполнения запросов. 
	- Traffic shifting: пошаговая маршрутизация трафика между версиями сервисов (canary strategy, blue-green strategy of deployment, A/B-тестирование).
