========== selector ================
Selectors — это механизм, который позволяет идентифицировать ресурсы на основе меток.  Есть два типа селекторов: 
На основе равенства 		— фильтруют ресурсы по ключам и значениям меток. Например: key=value — выбирают ресурсы с ключом метки, равным значению, key!=value — ресурсы с ключом метки, не равным значению.
На основе набора 		— позволяют фильтровать ресурсы по ключам, которые принадлежат к набору значений. Поддерживают операции in, notin и exists.

========== labels ==================
Labels — это пары «ключ-значение», которые можно прикрепить к объектам Kubernetes, таким как:
	узлы (Nodes), 
	поды (Pods), 
	объёмы (Volumes). 
	сервисы (Services), 
	деплойменты (Deployments) и другие

----------- Польза -----------------------
- это основной способ группового управления ресурсами в Kubernetes.
В кластере или namespace много микросервисов со своими функциями (back, front, etc.) нужно сделать метку для приложения или для объекта приложения. Например: сделать несколько меток для нескольких объектов одного и того же микросервиса. Например: метка = имени микросервиса.
Cluster controllers могут управлять метками, чтобы была возможность:
	- по labels выделить, чтобы через selectors управлять объектами (масштабировать, удалять и т.д.);
	- найти все объекты кластера по метке;
	- как будут фильтроваться, при тех или иных условиях, сущности микросервиса.


========== Сущности для управления размещением PODs на NODS =====================
NodeSelector			- простая фильтрация
NodeAffinity / Anti-Affinity - продвинутая принадлежность/анти-принадлежность
	NodeAffinity	- гибкие правила принадлежности
	PodAntiAffinity	- правила разделения Pod'ов (анти-принадлежность)
Taints/Tolerations		- ограничения доступа к нодам:
	На ноде (Taint)
	В Pod (Toleration)


========= Logs & events ==========================================================
Логи живут пока ПОД работает. Для постоянного лигирования требуются сторонние сервисы.
Логи приложения пишутся в контейнер.,
Логи кластера можно посмотреть в evets:
Kubernetes Events 		— это временные записи (живут пару часов), которые предоставляют информацию о значимых событиях или изменениях в кластере Kubernetes. Они помогают отслеживать статус и здоровье ресурсов кластера, представляют различные этапы жизненного цикла объектов Kubernetes (подов, сервисов, развёртываний и других). Особенности:
	-	События не хранятся постоянно, обычно имеют время жизни (TTL). После превышения TTL событие автоматически удаляется.
	-	Каждый контроллер Kubernetes генерирует события, чтобы сигнализировать о изменениях или проблемах в своих управляемых ресурсах.
События в Kubernetes можно разделить на два типа: 
	Normal — указывают на ожидаемые и рутинные операции в кластере, например, планирование пода или успешное вытягивание образа.
	Warning — указывают на проблемы, которые нужно устранить. Например, неудачное планирование пода, ошибки вытягивания образа или проблемы с лимитами ресурсов.


======== Resources ================================================================
для каждого контейнера, каждого пода задать ограничения, чтобы не было конфликтов в кластере:
	CPU	1ядро = 1'000 мили ядер;
	Memory
deployment.yaml
spec:
      containers:
        - name: hw2-app
          image: tonnelier/my-app-2-hub:v2
          ports:
          - containerPort: 8080
          env:
          - name: ENVIRONMENT      # ← добавить name:
            value: "learning"      # ← и value:
          resources:			# то, что мы хотим взять у кластера. Сколько будет выделено кластером для контейнера:
            requests:
              memory: "100Mi"
              cpu: "100m"		# kernel работает 1 раз в 10 тактов частоты.
            limits:			# ограничение для данного ресурса. Больше кластер не выделит ресурсов:
              memory: "500Mi"
              cpu: "500m"		# 1/2 kernel
Нужно понимать, что заявленные ресурсы кластера бронируют ресурсы в кластере и их может не хватить другим микрочервисам. Это можно увидеть в системе мониторинга.
Реальный расход памяти и процессорной ёмкости можно увидеть через команды kubctl.

1-Container + 1-Sidecar (один или несколько) = 1-Pod
-------- Sidecar-контейнер в Kubernetes -------------------
— это вспомогательный контейнер, который работает вместе с основным приложением в одном модуле (Pod). Цель использования sidecar-контейнеров — расширение функциональности основного приложения за счёт дополнительных сервисов, таких как (служебная информация):
- ведение логов;
- трассировка;
- мониторинг;
- безопасность или синхронизация данных. При этом код основного приложения не модифицируется напрямую. 
Цель использования sidecar-контейнеров — расширение функциональности основного приложения за счёт дополнительных сервисов, таких как ведение логов, мониторинг, безопасность или синхронизация данных. При этом код основного приложения не модифицируется напрямую.

======== Probes ==============================================================
startupProbe:		для тяжеловесных приложений. интервал опроса и частота проб. Можно указать команду для опроса.
readinessProbe:		показывает: готово ли приложение к работе.
livenessProbe:	 	проверят доступность контейнера.
-------- параметры:
periodSeconds:		частота проверки контейнера.
timeoutSeconds:		время ожидания ответа от пробы.
initialDelaySeconds:	как часто запускать пробы.
failureThreshold:	количество повторных попыток до пометки пробы как неудачной
